## Final Architecture Decisions

### SDK Integration

| SDK | Package | Usage |
|-----|---------|-------|
| Microsoft Agent Framework | `agent-framework[foundry,orchestrations]` | Agent creation, tool decorator, handoff workflow |
| Azure AI Foundry SDK v2 | `azure-ai-projects` | Optional agent registration via `agents.create_version()` |
| Azure Identity | `azure-identity` | `AzureCliCredential` for local dev, `DefaultAzureCredential` for registration |

### Agent Design

- **`FoundryChatClient.as_agent()`** — Creates workflow-compatible `Agent` instances from a shared chat client
- **Instruction-based behavior** — Each agent has a dedicated instruction prompt stored as a module-level constant (e.g., `TRIAGE_INSTRUCTIONS`)
- **Tool attachment** — Tools are passed directly to `as_agent()` via the `tools=` parameter

### Workflow Pattern

- **`HandoffBuilder`** — Declarative bidirectional handoff routing
- **Start agent** — Triage is always the entry point
- **Streaming** — `workflow.run(input, stream=True)` yields events for real-time output

### Error Handling Strategy

- **Typed exceptions** — `ChatClientInvalidAuthException`, `WorkflowException`, `AgentFrameworkException` instead of bare `Exception`
- **Tool-level validation** — Input guards in `process_refund` (amount > 0, order exists, amount ≤ total)
- **Graceful degradation** — User-friendly error messages with actionable guidance (e.g., "Run `az login`")

### Security

- **Human-in-the-loop** — `approval_mode="always_require"` on `process_refund`
- **No hardcoded credentials** — All secrets via environment variables
- **Defensive tool design** — All edge cases (missing orders, invalid amounts) return structured JSON errors
